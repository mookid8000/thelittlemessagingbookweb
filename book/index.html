Idempotency

By now, itâ€™s probably clear that â€œat least onceâ€-delivery is preferred, because that is the only delivery guarantee that allows us to be sure that we get to handle each message and not risk losing one.

Whatâ€™s left now is to ensure that the world (i.e.Â our system) does not end up in the wrong state if we get to experience a re-delivered message. So how do we do that?

In math, an â€œidempotentâ€ operation is a function that yields the same result, no matter how many times it is called with the same input. We borrow this word, and then we aim for making our message handling idempotent!

How we exactly go about doing that certainly depends a lot on which kind of work we are doing.

In most cases, we generate a bunch of side effect that we can consciously choose to ignore.

If e.g.Â your message handler outputs log statements to a set of files or a log aggregator, then the fact that the log statements go into the files or the aggregator is a side effect. Message ordering

Even though most (if not all) queueing systems are FIFO, i.e.Â they adhere to the â€œfirst in first outâ€ principle, you should not count too much on it, as thereâ€™s several normal and good reasons why messages can be more or less reordered.

To achieve higher message processing rates, itâ€™s pretty normal to allow for some kind of parallelism when processing incoming messages. Lastly, itâ€™s pretty normal for message handling to sometimes fail, which ultimately ends up moving the problematic message to a dead-letter queue. In summary: Build your systems with tolerance for some degree of message reordering, maybe even to tolerate messages in any order. If you think of each message as an operand in algebra, you could say that systems need to be commutative.

Another design decision made in Rebus is that you should not handle (most) errors, you should let exceptions occur.

Queues vs.Â logs

From afar, queue-based brokers and log-based brokers may seem interchangeable, and in theory you could model one with the other â€“ but they have a bunch of differences that would make it difficult and pretty wonky outside of an academic setting to try to switch from queues to logs or from logs to queues in most realistic scenarios!

Ordering

Log-based brokers provide a strict guarantee of the ordering of messages, messages are not deleted as they are read, and consumers themselves keep track of which messages they have read, which they can easily do because messages are numbered sequentially. To distribute work among multiple processes, log-based brokers offer the concept of â€œconsumer groupsâ€, which means that the broker helps with distributing messages among multiple consumer instances. Because of the strict ordering, if a log-based consumer encounters an error (e.g.Â a NullReferenceException ğŸ˜…), it has no other choice than to stop processing messages if ordering must be preserved. Queue-based consumers will usually receive and handle one single message at a time, and then concurrency is used to increase throughput by increasing parallelism, but the individual consumer deals with a single message at a time.
